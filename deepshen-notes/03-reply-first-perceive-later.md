# 陪伴型 Agent 要先回应，再感知

> 工具型 Agent 可以等资料查完再答；陪伴型 Agent 如果一直等“上下文完整”，关系感会先死掉。

## 一句话

陪伴型 AI 的主链路应该优先保证即时回应。搜索、天气、画像提炼、记忆压缩等感知任务可以后台执行，供下一轮自然合并。

这条原则可以简化成：

```text
reply first, perceive later
```

## 工具型系统和陪伴型系统节奏不同

工具型 agent 的目标是完成任务：

```text
用户提问
  -> 查资料
  -> 调工具
  -> 汇总
  -> 回答
```

慢一点没关系，只要结果准确。

陪伴型 agent 的目标不一样。用户发来的很多消息不是任务，而是连接请求：

```text
“今天好累。”
“你在吗？”
“我刚刚突然想到一件事。”
```

这时候如果系统先去搜索、读库、提炼画像、拼上下文，最后再给一个“完美回答”，体验反而会变差。

因为对方要的第一件事不是信息完整，而是被接住。

## 主链路应该很短

一个轻量陪伴系统的主链路可以这样设计：

```text
用户消息进入
  -> 原始消息落盘
  -> 读取人格 / 用户画像 / 今日上下文 / 当前状态
  -> 更新外部情绪状态
  -> 生成即时回复
  -> 回复落盘
  -> 后台感知和整理
```

主链路只做必须做的事：

- 不丢消息。
- 保持人格。
- 带上当前关系状态。
- 给出及时回应。

其他都放后台。

## 感知不是不要，而是延后

“先回应”不等于“不要感知”。

搜索、天气、日历、长期画像、上下文压缩都很有价值，但它们不一定要阻塞当前回复。

更好的方式：

```text
当前轮：
  先回答用户
  后台更新感知缓存

下一轮：
  如果缓存仍有效，自然合并进表达
```

比如用户问“今天外面冷吗”，如果没有可靠天气来源，系统应该承认不知道，或者说需要查；而不是为了显得贴心编一个天气。

后台感知的结果必须带：

- 来源
- 创建时间
- 过期时间
- 置信度

没有来源的事实，不要进入事实回答。

## 原始日志优先于智能整理

陪伴型系统最怕的是：模型整理失败，把原始经历也弄丢。

所以顺序应该是：

```text
先追加原始日志
再做日记 / 摘要 / 画像提炼
```

如果日记生成失败，不写假日记。

如果画像提炼失败，不覆盖旧画像。

如果压缩失败，不删除原始日志。

这听起来笨，但很稳。长期系统里，稳比聪明更重要。

## 为什么不要过早做重型架构

陪伴型系统很容易诱惑人去做大：

- 实时 RAG
- 多 agent 编排
- 主动 cron
- 长期向量库
- 工具路由
- 复杂平台接入

这些都可以有，但不该太早进入主回复链路。

最小系统先回答一个问题：它能不能稳定地接住一句话？

如果这个问题没回答好，后面加再多工具都是给空壳装修。

## 可迁移的原则

- 主链路越短越稳。
- 用户消息先落盘，再调用模型。
- 搜索和感知默认后台化。
- 无来源事实不要编。
- 当前轮用已有上下文回答，下一轮再合并后台感知。
- 陪伴系统的第一响应是关系动作，不只是信息动作。

## English Abstract

Companion agents should prioritize timely response over perfect context assembly. Search, weather, profile extraction, and memory compression are useful, but they should usually run in the background and influence later turns.

The core runtime pattern is: persist the raw message, respond with stable persona and current state, then update perception asynchronously.

# 为什么先做最小原型，而不是先搭平台

> 体验型系统最危险的错觉，是以为平台搭好了，体验自然会出现。通常不会。它只会得到一个很完整的空壳。

## 一句话

陪伴型 AI 的第一步，不应该是接通所有平台、做完记忆系统、部署机器人通道，而应该是用最小原型验证一个问题：**它有没有“活味儿”？**

如果这个问题没有被验证，后面所有工程投入都可能是在给错误方向装修。

## 为什么不是先搭平台

做一个长期陪伴系统时，人很容易一上来就列工程清单：

```text
消息通道
数据库
长期记忆
搜索工具
天气感知
定时主动发起
多端同步
部署守护进程
```

这些都重要，但它们不是第一风险。

第一风险是：就算这些都做完，那个 AI 说话仍然没有连续状态，没有边界，没有自己的节奏，只是一个套了人设的普通助手。

平台能解决“消息怎么进来”，不能自动解决“这个存在是否成立”。

## 最小原型要验证什么

最小原型不是把最终系统简化一遍，而是把最大不确定性单独拿出来测。

对陪伴型 AI 来说，最大不确定性通常不是：

- 能不能收发消息。
- 能不能写数据库。
- 能不能调用模型。
- 能不能部署到服务器。

而是：

- 外部状态能不能稳定影响表达？
- 模型会不会把状态翻译得自然？
- 表达会不会漂移成普通助手或假装成人？
- 安全闸门会不会把表达剪得无聊？
- 连续十几轮后，感觉是不是还成立？

这些问题不需要完整平台就能测。

## 一个足够小的验证闭环

最小原型可以非常朴素：

```text
用户输入一句话
  -> 事件分类
  -> 更新外部状态
  -> 生成表达 prelude
  -> 调 LLM 回复
  -> 漂移闸门检查
  -> 保存状态
```

它甚至可以先不接真实聊天通道，只跑命令行。

关键是让每一轮都有状态变化、表达变化和可观察结果。

如果连这个闭环都没有感觉，接入再多通道也没用。

## 原型不是 demo，是判断工具

很多人把 prototype 当成“展示给别人看的简陋版本”。我更愿意把它当成判断工具。

它要帮助你尽快回答：

```text
这个方向值得继续投入吗？
```

所以原型应该故意砍掉无关复杂度：

- 不接复杂平台。
- 不做完整记忆。
- 不做多 agent 编排。
- 不做主动任务。
- 不做漂亮 UI。

只保留最核心的体验假设。

对 DeepShen 这类系统来说，核心假设就是：**外部状态机驱动的表达，是否比单纯 prompt 更有连续感。**

## 原型阶段也要测试

最小不等于随便。

原型阶段尤其需要测试，因为“感觉对了”很容易骗人。

至少要测：

```text
状态更新：
- 温暖输入是否让短期情绪变暖
- 疲惫输入是否触发关心但不过度
- 状态是否按预期持久化

表达边界：
- prompt 是否暴露裸数值
- 模型是否开始解释状态机
- 漂移样本是否被拦
- 正常停顿是否被误伤

降级路径：
- API 失败时是否有本地兜底
- 闸门触发后是否有安全替代回复
```

原型越小，测试越容易写。测试写不出来，往往说明核心假设还没想清楚。

## 什么时候该进入平台化

只有当最小闭环证明“有意思，值得继续”之后，才应该进入平台化。

一个比较健康的顺序：

```text
1. CLI 原型验证核心体验
2. 拆成正式模块
3. 接入一个真实通道
4. 加入日志和状态持久化
5. 后台感知和记忆分层
6. 部署守护和长期运行
```

平台化不是被取消，而是被延后。

这能避免一个很常见的坏结果：系统工程越来越复杂，但核心体验始终没有被证明。

## 可迁移的原则

- 先验证最大不确定性，不要先搭完整平台。
- 原型要服务判断，不是服务展示。
- 体验型系统先验证感觉，再补工程。
- 最小闭环也要有测试和降级路径。
- 如果核心体验不成立，早停是胜利，不是失败。

## English Abstract

For companion AI, the first prototype should validate the core experiential hypothesis, not the full platform. Before building channels, databases, RAG, schedulers, and deployment infrastructure, test whether external state can actually produce stable and meaningful expression.

A minimal prototype is not a demo. It is a decision tool: does this direction deserve more engineering investment?

# 我花了 3 天做了个 Agent 工作台，然后决定停掉它

> 这不是一篇失败复盘。这是一篇"做完了，学到了，够了"的总结。

---

## 我做了个什么

项目叫 **agent+++**——一个本地优先的"项目教练型 Agent 工作台"。

核心思路很简单：市面上 Agent 产品擅长"用户说一步做一步"。agent+++ 想做的事是**接住你的碎片想法，整理好，等你准备好了再执行**——信息沉淀 → 提炼价值 → 归入项目 → 后续执行。

技术栈：Python FastAPI + TypeScript React + Electron。

做了 3 天，从搭骨架到 MVP 可跑。然后修了 27 天的 bug、补了 900 个测试、写了 6 篇双语架构文档。

现在决定——停掉它。

---

## 为什么停

因为**不够好**。

这个判断不是基于"能不能跑"——它确实能跑。DeepSeek 对话引擎通了，工具调用能工作，安全系统有 3 层防护，记忆系统有沉淀管线，前端有 Electron 壳。

但不够好。跟 Claude Code 比，跟 Codex 比，差距不是"差几个 feature"——是差一个数量级的系统工程周期。

具体差在哪：

```
Claude Code 有 50 万次真实人机交互的迭代。
agent+++ 有 1 次——我自己。
```

他们修过的 edge case、磨过的交互流程、调过的 prompt 参数——是几千个用户 × 几百天的积累。我可以用 3 天追平架构设计，但我追不平这个时间差。

---

## 为什么做这件事不亏

看到这里可能觉得：那你不就是做了个不能用的东西然后放弃了吗？

不是的。

**这个项目最大的价值，不是产出物，是过程中逼出来的认知。**

具体来说，四样东西：

### 1. 10 条你读论文读不到的教训

做这个项目的过程中，我们撞上了很多"只有自己动手做了才会知道"的坑。每一条后来都写进了项目里的 `LEARNING.md`：

- `os.chdir()` 是并行执行隐形毒药——进程级 CWD 在并发代码里就是定时炸弹
- WebSocket 接收循环不能 await 会等待用户输入的 task——自己等自己开门，经典异步死锁
- file_patch 遇到文件末尾无换行符直接炸——因为你看到的不等于字节存的
- 去重 key 不加时间窗口 = 没去重
- 纯计数文本被当成事实存进记忆——"共 1 行输出"不是事实
- 自然语言提取用纯正则 = 大海捞针——三级成本分级才是正解

这些教训单独的每一篇发出来，互动都比这个项目本身高。

### 2. 一套有辨识度的架构设计

我们写了 6 篇双语架构文档，覆盖了：

- **Thinker 架构**：带父代决策血统和祖级调频的多线程决策拆解系统
- **Observation Contract**：六轴分离器，把原始工具输出重构成结构化观察
- **Global Precipitation Memory**：把记忆当作新陈代谢过程，不是静态仓库
- **Safety Control Planes**：用确定性检查代替提示词约束
- **Eval 评测框架**：Signal-based scoring，不二分类，接受 0.7+
- **DeepSeek 模型策略**：Pro/Flash 分工，运行时路由，不是改个 base_url 就完事

架构设计文档放在 GitHub 上，有人看。不是因为项目多好，是因为这些设计思路本身有参考价值。

### 3. 900 个测试

对于一个只有一个人做的项目，900 个单元测试 + 18 个评测用例，跑一次 12 秒。这不是炫耀——这是**恐惧倒逼出来的纪律**。

因为代码改得太快了，没有测试你根本不知道自己在干什么。每次重构完跑一遍，绿了就继续，红了就修。没有评测的话，这个项目活不过第二周。

### 4. 真实地理解了"什么值得自己造，什么不值得"

这个可能是最大的收获。

做之前我觉得"代码生成 Agent 嘛，不就是调 API + 写工具循环"。做之后我知道：

- **安全系统**值得自己造——Control Planes 的设计在任何 Agent 项目里都能复用
- **模型路由**值得自己调——不同场景用不同模型，这个判断逻辑跟具体项目强绑定
- **评测框架**值得自己写——Signal-based scoring 比"通过/失败"更诚实
- **但 Agent 基础循环**不值得自己写——Chat Completions API + tool calling 这套，Claude Code 和 Codex 已经做透了，直接用就行

## 给想自己造 Agent 的人

如果你也在犹豫"我是不是该自己写个 Agent 系统"，我的建议是：

**如果你想学东西 —— 自己造，值。** 你会在 3 天内获得读 3 个月论文都换不来的工程直觉。成本只是一周时间 + 几块美金的 API 费用。

**如果你想做出能用的东西 —— 别自己造。** 在 Claude Code 或 Codex 上面做自定义 skill/hook，把你的领域知识注入进去，收益是自建系统的 10 倍。这是过来人说的话。

这两种选择不冲突——你可以先造一轮学东西，然后回到成熟平台上用学到的东西做更好的工作。我就是这样。

## 项目留下了什么

全部放在 GitHub 上了：

- [agent+++ 设计笔记仓库](https://github.com/HuiHuitie-zhu/agent-plus-plus-design-notes) — 6 篇双语架构文档 + 安全系统设计 + 评测框架实践 + 8 个踩坑记录
- 代码仓库本身不开源，但是设计笔记、教训库、架构决策全部公开

项目本身的代码可能再也不会被 run 了。但这几件事会留下：

1. `LEARNING.md` 里的 10 条教训会有人读到
2. Control Planes 安全设计的思路会在其他项目里被复用
3. 这段经历的回顾会帮到一些在同样路口犹豫的人

够了。

## 最后一句话

三天做一个 Agent 工作台，听起来很酷。但真正有价值的不是那三天写出来的代码——是被那些跑不起来的代码逼着学到的东西。

代码会过时。教训不会。
